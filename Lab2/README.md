### Експеримент 1.1: Naive Dockerfile

**Файл:** `deploy.lab-containers-starter-project-python/Dockerfile.naive`

**Особливості:**
- Базовий образ: `python:3.13-slim`
- Копіювання всіх файлів одразу
- Встановлення залежностей після копіювання коду

**Команда збірки:**
```bash
docker build -f Dockerfile.naive -t spaceship:naive .
```

**Результати:**
- **Розмір образу:** ~300 MB
- **Час першої збірки:** 30 секунд
- **Час пересбірки (зміна коду):** 18 секунд

**Проблема:**
- При зміні будь-якого файлу коду (наприклад, `spaceship/app.py`) інвалідується шар з залежностями
- Pip переінсталює всі пакети при кожній зміні коду
- Неефективне використання кешу Docker

### Експеримент 1.2: Optimized Dockerfile (оптимізація послідовності)

**Файл:** `deploy.lab-containers-starter-project-python/Dockerfile.optimized`

**Особливості:**
- Спочатку копіюємо `requirements.txt`
- Встановлюємо залежності в окремому шарі
- Копіюємо код останнім

**Стратегія шарів:**
1. Копіювання `requirements.txt` (змінюється рідко)
2. Встановлення залежностей (кешується)
3. Копіювання коду (змінюється часто)

**Результати:**
- **Розмір образу:** ~300 MB (аналогічно naive)
- **Час першої збірки:** 30 секунд
- **Час пересбірки (зміна коду):** **2,5-3 секунди**

**Покращення:**
- **Пришвидшення пересбірки у 8 раз**
- Залежності кешуються та не переінсталюються
- Змінюється тільки шар з кодом

**Експеримент з додаванням коментаря:**

Додано в build/index.html
```html
<h2>Maluiev Pavlo IM41 Lab 2</h2>
```

Пересбірка: **3 секунди** замість 18 секунд у naive версії.

### Експеримент 1.3: Debian vs Alpine базові образи

#### Debian-based образ

**Файл:** `deploy.lab-containers-starter-project-python/Dockerfile.debian`

**Особливості:**
- Базовий образ: `python:3.13-bookworm` (Debian 12)
- Містить glibc
- Більше системних утиліт

**Результати:**
- **Розмір образу:** ~1.6 GB
- **Час збірки:** 23 секунди
- **Час збірки з numpy:** 30 секунд

**Переваги:**
- Сумісність з більшістю Python пакетів
- Наявність wheels для популярних бібліотек (numpy, pandas)
- Швидка установка залежностей (pre-compiled)

**Недоліки:**
- Більший розмір образу
- Більше потенційних вразливостей

#### Alpine-based образ

**Файл:** `deploy.lab-containers-starter-project-python/Dockerfile.alpine`

**Особливості:**
- Базовий образ: `python:3.13-alpine`
- Містить musl замість glibc
- Мінімальний набір системних пакетів

**Результати без numpy:**
- **Час збірки:** 22 секунд

**Результати з numpy:**
- **Розмір образу:** ~202 MB
- **Час збірки:** 37 секунд
- Потрібні build-залежності: `gcc musl-dev linux-headers`

**Порівняння Alpine vs Debian з numpy:**

| Метрика | Alpine | Debian | Різниця |
|---------|--------|--------|---------|
| **Розмір образу** | ~202 MB | ~1600 MB | **-792%** |
| **Час збірки** | 37s | 30s | **+81%** |
| **Build залежності** | Потрібні | Не потрібні | - |

**Чому Alpine повільніший з numpy:**
- Відсутні pre-compiled wheels для musl
- Numpy компілюється з вихідників
- Потрібна компіляція C-розширень (BLAS, LAPACK)
- Тимчасові build-залежності займають додатковий простір

### Експеримент 1.4: Додавання numpy та matrix endpoint

**Зміни в проекті:**

1. Додано `numpy` в `requirements.txt`
2. Створено новий endpoint

```python
import numpy as np

@app.get("/matrices")
    def get_matrices() -> dict[str, list[list[float]] | str]:
        matrix_a = np.random.rand(10, 10).tolist()
        matrix_b = np.random.rand(10, 10).tolist()
        product = np.matmul(np.array(matrix_a), np.array(matrix_b)).tolist()
        
        return {
            "student": "Pavlo Maluiev",
            "student_id": "14840136",
            "matrix_a": matrix_a,
            "matrix_b": matrix_b,
            "product": product,
            "dimensions": "10x10"
        }
```

**Висновки про вплив numpy:**
- На Debian: майже непомітне збільшення часу збірки
- На Alpine: час збірки зростає у **~2 рази**
- Розмір образу зростає на ~70 MB

## Частина 2: Golang Application

### Експеримент 2.1: Single-Stage Build (базова лінія)

**Файл:** `deploy.lab-containers-starter-project-golang/Dockerfile.single`

**Підхід:**
- Один етап збірки
- Залишаємо Go toolchain в фінальному образі
- Компіляція та виконання в одному образі

**Результати:**
- **Розмір образу:** ~500 MB
- **Час збірки:** 14.5 секунди (перша збірка)
- **Час пересбірки:** 2 секунди (з кешем)

**Аналіз вмісту образу:**
- Скомпільований бінарний файл (~17 MB)
- Go compiler та toolchain 
- Go standard library sources
- Build cache та модулі 
- Системні утиліти 

**Що потрібно для запуску:** Тільки бінарний файл (< 4% від образу)  
**Що НЕ потрібно:** Все інше (~96% образу)

**Переваги:**
- Простота налаштування
- є shell

**Недоліки:**
- 98% простору витрачається даремно
- Повільніше завантаження образу
- Вищі витрати на зберігання

### Експеримент 2.2: Multi-Stage Build з FROM scratch

**Файл:** `deploy.lab-containers-starter-project-golang/Dockerfile.scratch`

**Підхід:**
- **Етап 1 (builder):** Збірка статичного бінарного файлу
- **Етап 2 (runtime):** Копіювання тільки бінарника в порожній образ

**Особливості збірки:**
- `CGO_ENABLED=0` - вимкнення CGO для статичного лінкування
- `-ldflags '-extldflags "-static"'` - форсування статичного лінкування
- `FROM scratch` - буквально порожній образ (0 байт)

**Результати:**
- **Розмір образу:** ~17,2 MB
- **Час збірки:** 10 секунд
- **Скорочення розміру:** **~96%** порівняно з single-stage

**Аналіз вмісту:**
- Тільки скомпільований статичний бінарний файл

**Переваги:**
- Мінімально можливий розмір образ

**Недоліки та проблеми:**
- Неможливо зайти в контейнер (`docker exec` не працює - немає shell)
- Немає інструментів для дебагу (ls, cat, curl, тощо)

**Коли використовувати:**
- Критична мінімізація розміру

---

### Експеримент 2.3: Multi-Stage Build з Distroless

**Файл:** `deploy.lab-containers-starter-project-golang/Dockerfile.distroless`

**Підхід:**
- **Етап 1 (builder):** Збірка оптимізованого бінарного файлу
- **Етап 2 (runtime):** Distroless base образ

**Особливості:**
- Базовий образ: `gcr.io/distroless/static-debian12`
- Оптимізація: `-ldflags="-s -w"` (видалення debug символів)
- Запуск від non-root користувача

**Результати:**
- **Розмір образу:** ~18 MB
- **Час збірки:** 10 секунд
- **Скорочення розміру:** **~97%** порівняно з single-stage

**Аналіз вмісту:**
- Скомпільований бінарний файл
- CA certificates (HTTPS працює)
- Timezone database (tzdata)
- `/etc/passwd` (non-root user)
- Мінімальний runtime (glibc)
- Немає shell
- Немає package manager

**Переваги:**
- Майже такий же малий, як scratch (+2-5 MB)
- HTTPS працює "з коробки"
- Timezone operations працюють коректно

**Недоліки:**
- Все ще немає shell (не можна exec для дебагу)

**Коли використовувати:**
- **Production deployments** 
- Сервіси з HTTPS запитами
- Баланс між розміром образу та функціональністю

## Ключові висновки

### 1. Оптимізація шарів Docker критично важлива

**Виявлено:**
- Порядок команд COPY в Dockerfile впливає на швидкість пересбірки у **8-10 разів**
- Залежності (requirements.txt, go.mod) змінюються рідко → копіюємо першими
- Код змінюється часто → копіюємо останнім

```dockerfile
# Правильно
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .

# Неправильно
COPY . .
RUN pip install -r requirements.txt
```

### 2. Вибір базового образу - це завжди компроміс

**Alpine для Python:**
- ✅ **Значно менший розмір** (~202 MB vs ~1.6 GB Debian, **-87%**)
- ❌ Трохи повільніша збірка з numpy (37s vs 30s, **+23%**)
- ❌ Musl libc - компіляція з вихідних кодів для C-розширень
- ⚠️ Відсутні pre-compiled wheels для numpy/pandas
- ✅ **Рекомендовано** якщо критичний розмір образу

**Debian для Python:**
- ✅ Швидка збірка (pre-compiled wheels)
- ✅ Широка сумісність (glibc)
- ❌ **Дуже великий розмір** (~1.6 GB з numpy)
- ✅ Краще для складних ML/Data Science застосунків
- ⚠️ Тільки якщо розмір не критичний

**Scratch/Distroless для Golang:**
- ✅ Мінімальний розмір (17-18 MB замість 500 MB)
- ✅ Максимальна безпека (мінімальна поверхня атаки)
- ❌ Складний дебаг (немає shell та інструментів)
- ✅ **Обов'язково для production**

### 3. Multi-stage builds обов'язкові для compiled languages

**Виявлено:**
- Golang: **96-97% скорочення розміру** (500 MB → 17-18 MB)
- Build інструменти не потрібні в runtime
- Нульовий вплив на продуктивність виконання
- Фінальний образ містить тільки бінарний файл

**Рекомендація:**
- Development: single-stage (зручність дебагу)
- Production: multi-stage з distroless (безпека + розмір)

### 4. Оптимізація шарів дає найбільший практичний ефект

**Виявлено:**
- Python: пересбірка **18s → 2.5-3s** (покращення у **6-7 разів**)
- Простий переупорядок команд COPY
- Критично для швидкості CI/CD
- Залежності кешуються, змінюється тільки код

## Висновки

### Для Python застосунків

1. **Завжди оптимізуйте порядок шарів:**
   - Спочатку `COPY requirements.txt`
   - Потім `RUN pip install`
   - В кінці `COPY` код застосунку

2. **Вибір базового образу:**
   - **Прості застосунки (FastAPI, Flask):** `python:3.x-alpine`
   - **З numpy/pandas/ML бібліотеками:** `python:3.x-slim` (Debian)

3. **`.dockerignore`:**
   ```
   __pycache__
   *.pyc
   .git
   .pytest_cache
   venv/
   ```

### Для Golang застосунків

1. **Завжди використовуйте multi-stage builds в production:**
   - Single-stage тільки для development
   - 98% економії простору без недоліків

### Для CI/CD

1. **Вимірюйте розміри образів**
2. **Використовуйте specific tags:** не `latest`, а `python:3.13-slim`
